Introduction

In the current field of high-performance computing (HPC), compilers serve as the fundamental bridge between high-level programming languages and the low-level operations executed
by hardware. They provide the essential infrastructure for performance-critical tasks, such as
instruction scheduling, register allocation, and hardware-specific code generation. Within the
compiler ecosystem, LLVM [1] has emerged as a predominant framework for modern compiler
construction, underlying widely used toolchains for languages such as C/C++ [2], Rust [3], and
Swift [4]. This widespread adoption is attributed to its modular architecture, its support for a
diverse range of hardware targets, and its extensive suite of optimization passes.
The LLVM framework is under constant development, with frequent changes that present significant engineering challenges. The project maintains a rapid release schedule; over the past
year alone, there have been two major version cycles and approximately 15 canonical releases.
Because optimization algorithms evolve with each of these versions, these shifts can lead to
unpredictable fluctuations in both execution speed and memory usage for scientific applications—sometimes even resulting in regressions. Despite the potential for such shifts, the industry still lacks a robust, reliable tool to track and record these comprehensive performance trends
over the long term.
To solve these problems, this project plans to design and build a performance testing system
that is easy to use and highly automated. It is specially made to compare and check how performance changes across different LLVM versions. The system will use the Lit [5] tool inside
LLVM to run the tests. It will focus on two very typical test suites: the official LLVM test suite
and the RAJA performance test suite [6], which is for C++ and supports various parallel programming models such as OpenMP [7] and CUDA [36]. By building a system that automatically
downloads source code, finishes the compilation, runs the tests, analyses the results, and turns
them into visualisations, this project can provide a serious and useful tool for researchers and
developers when they need to evaluate and pick a compiler version.

2

Background and Literature Review

In the current field of high-performance computing, the LLVM compiler framework is used very
widely [8, 9, 10]. It uses a very clear “three-stage modular design,” which means the whole
compilation process is divided into frontend parsing, middle-end optimization, and backend machine code generation. This design makes the system very easy to expand. However, as the
compilation process gets more complex, it becomes harder to accurately track its performance
[11, 12, 13]. The LLVM project follows a very regular release schedule, putting out a major
version about every six months. Because compiler optimization basically uses a set of rules
to find a relatively good solution in a very large search space, any change to the compiler’s
internal structure or logic can cause the running speed and memory usage of scientific applications to show clear and irregular fluctuations. Some actual studies have proven that version
updates do not always mean the execution efficiency gets better. Some deep optimizations
made for specific code patterns might accidentally cause performance to drop for other types of
tasks. Therefore, it is hard to accurately evaluate a compiler’s actual performance just by using
static analysis methods. Comparing running performance across versions in a real hardware
environment is necessary.
To monitor how compiler performance changes over time, the industry and the development
community have released many testing tools and frameworks. However, if we want to quickly

1

compare and analyze different compiler versions on a single node, these existing solutions often
show some limitations:
• Lit is a lightweight testing tool that comes with the LLVM system. It is mainly used to run
a large number of regression tests at the same time [5]. However, its design is mainly
as a test execution engine. It did not consider the need for tracking performance over
a long period. It cannot automatically collect and summarize test data, and it does not
have a special storage mechanism to record and compare performance changes across
historical versions.
• The LLVM community developed the LNT system specifically to monitor compilation time
and execution efficiency [14]. But the architecture of this system is quite complex, and
the public test servers often run unstably. Also, it is not very good at integrating new test
requests. Most importantly, the user experience is poor, so developers in the community
are not very willing to use it. It is hard to meet the needs for quick verification [15].
• HPC Testing frameworks like ReFrame [16] or JUBE [17]. ReFrame provides a Pythonbased framework to automate the build, execution, and sanity checking of benchmarks
across diverse system partitions, while JUBE utilizes XML-based templates to manage
complex parameter studies and systematically extract performance data from log files.
While frameworks like ReFrame and JUBE offer robust features for large-scale systems,
their steep learning curve and complex XML/Python abstractions can be prohibitive for
individual researchers [18, 19]. For localized tasks such as LLVM performance evaluation
on a single workstation, the system overhead and deployment complexity of these "heavy"
frameworks often outweigh their benefits [20]. Since this project evaluates LLVM compiler
performance on a single machine, using these heavy frameworks adds unnecessary system overhead and makes deployment harder.
Based on the research and analysis above, this project did not choose those overly complex
benchmarking systems. Instead, it uses Python scripts to achieve automated performance
tracking. It wraps Lit to run the LLVM test suite, while using native execution programs for the
RAJA suite. To ensure the test results are representative under limited resource conditions, the
system integrates two very core test suites:
• Official LLVM Test Suite: This is a basic tool for evaluating code generation quality. It
contains many representative open-source applications and test libraries specifically for
evaluating vectorization performance [21]. This suite is naturally compatible with the Lit
tool and supports a special testing mode, ensuring the data collected is consistent with
the optimization goals of the official LLVM community, which primarily focus on maximizing
execution performance, minimizing code size, and maintaining compilation efficiency [22].
• RAJA Performance Test Suite: Suggested by the project proposer, Michele Weiland
(EPCC), this suite is utilized because many modern high-performance computing apps
rely heavily on C++ template programming to build cross-platform abstraction layers [23].
This really tests the compiler’s ability to handle code inlining and remove redundant code.
Traditional general tests often fail to find defects in the compiler when it handles these
advanced features [24]. The RAJA suite is specifically used to measure the running efficiency of loop calculation kernels [6]. A loop calculation kernel is a computationally
intensive code segment, typically a loop, that performs the core numerical operations
of an algorithm and often dominates the overall execution time. It calculates a metric
called “Figure of Merit” (FOM), which compares the running time of raw code versus abstract code to measure the compiler’s optimization level [24]. If the level drops too much,
it means the compiler has a clear performance regression when handling modern C++
code [24]. These problematic code snippets can be given directly to compiler developers

2

for later fixing work [24].
Although this project currently focuses on the two test suites mentioned above, a complete
performance evaluation system needs to consider more dimensions. For example, OpenMPrelated benchmarks can be used to evaluate the overhead of the runtime library during thread
synchronization [7]. Additionally, the SOLLVE OMPVV test suite can be used to check if the
compiler’s meaning stays correct when implementing new programming standards [25].

3

Preliminary Investigations and Findings

This section explains the exploratory work and proof of concept done during the project’s preparation phase. So far, a complete automated process has been successfully built. This process
covers everything from getting the source code and compiling it automatically to running the
benchmark tests, analyses the results and showing the data in visualisations.

3.1

Developing an Automated Framework

The development of this project was done in a step-by-step way. In the beginning, LLVM source
code was configured, compiled, and tested by typing commands manually. This was done to
get a better understanding of how the underlying build infrastructure and its mechanics work.
After getting used to how the workflow works, these steps were put into basic Bash scripts so
the testing process could be run by executing a single command.
To make the system easier to maintain and expand, and to make sure the automation parts work
well with the data analysis and visualization code later, the automation framework rewritten into
Python. During this process, YAML [26] configuration files were added. This separates the logic
from the specific settings (like LLVM version numbers or build parameters), making the system
much more flexible [27].

3.2

Environment Control and Resource Optimization

Early compilation work was conducted in a local Ubuntu 24.04.3 LTS environment, and the
testing process was later moved to the EIDF VM, which runs Ubuntu 24.04.4 LTS.
During the process of compiling LLVM from source code, the system ran out of memory and
triggered a kernel error. At first, the EIDF virtual machine (4-core CPU and 4GB RAM) tried to
avoid this by limiting the number of parallel tasks, but this made a single compilation take 3838
seconds. Later, after upgrading the VM to a 16-core CPU and 32GB RAM, the compilation time
dropped to 1126 seconds, which made the testing much faster.
Also, to stop old CMake cache from making the compilation fail, a strict cleaning step was
added to the script. This forces the system to clean out old folders before every build. At the
same time, the system uses separate log files for different information, separating the progress
messages on the screen from the long compilation logs. This makes sure the process is easy
to see and can be checked later.

3.3

Core Technical Challenges and Solutions

When setting up the RAJA performance suite to check the OpenMP parallel model, there was
a linking error where the Clang [28] compiler—which serves as the C-family front-end for the
LLVM project—could not find the custom OpenMP library. To fix this, the pipeline was modified
to avoid relying on system-level environment variables for library resolution, which are unreliable
in this context. Instead, it embeds dynamic runtime search paths (using the RPATH parameter,
which embeds specific library search paths directly into the executable) into the linker flags.
This forces the benchmark to connect to the newly compiled LLVM library when the program
runs.
While running loop vectorization tests in the LLVM test suite, the system caught an unusual
floating-point error at runtime. By using the git bisect tool [37] to automatically search
through LLVM test suite code updates—as no issue had yet been reported on the LLVM GitHub
repository for the bug—it was found that the crash was caused by a bug recently added by
the main developers. This problem was fixed in a later patch (commit 488a96b). This process proved that this system is really able to find and locate problems in the underlying code
generation.

3.4

Data Parsing and Visualization Results

The official test suite mainly gives data in JSON [29] format, while the RAJA suite gives data
in comma-separated values (CSV) format. To handle all this data, the system has a standard
data extraction module. This module cleans the data from different sources and formats and
turns it into a common internal structure—a unified record that captures metadata (such as
suite and compiler versions, run IDs, and test status) and core performance metrics (including
execution/compile times, binary sizes, bandwidth, and FLOPS)—which is a good foundation for
later statistical analysis.
Currently, the work has covered several versions from LLVM 17.0.0 to 22.1.0. Based on the
cleaned data, the system uses the Plotly [30] library to make interactive web reports [31], as
shown in Figure 1.

Project Proposal

4.1

Core Research Questions and Hypotheses

In the field of high-performance computing, the optimization processes of modern compilers
are very complicated. This makes the performance of a compiler often change in irregular ways
when it is updated to a new version. This project aims to build and improve a testing framework
that is highly automated and can track LLVM compiler performance across different versions.
The core hypothesis of this project is: by building automated workflows, the system can compile
specific versions of LLVM whenever needed and run different benchmark suites (like the official
LLVM test suite and the RAJA performance suite). By cleaning and organizing the different
types of output data, the system can measure and show where the compiler’s performance
got better or worse as the architecture changed. This will give high-performance computing
developers solid, data-driven facts to help them choose which compiler version to use.

4.2

Minimum Viable Product

To make sure the graduation project is finished successfully, the final system must have at least
the following core modules. These modules are the basic requirements for the project:
• Full-process automated pipeline: Provide a stable automation engine implemented in
Python. This engine should use a YAML configuration file to specify a full loop of "getting
source code, building in an isolated environment, running tests, and saving logs" without
needing any manual help.
• Strict error handling and recovery: The system must be robust enough to handle errors.
If a certain compiler version fails to build or crashes while running, the system should not
just quit without a message. It must record the error accurately, safely skip that specific
task, and keep going with the rest of the testing queue.
• Data parsing adapters for different formats: Build a layer to extract data that can take
JSON data and CSV data from specific suites. It should clean this data and turn it into a
common internal format to make it easy to process later.
• Static benchmark report generator: Using the saved test data, the system should automatically create static HTML (with embedded JavaScript) charts. These charts will clearly
show core metrics like execution time and memory use for different test cases across
different compiler versions.

4.3

Extended Goals and Future Work

If the basic goals are finished, the project may expand further. The items below fall into two categories. Some — such as advanced visualisation, performance change detection, automated

5

version monitoring, and test subset selection — are scheduled as optional tasks in the workplan (Section 5) and will be attempted if time permits. The remaining items, including allowing
users to upload custom test sets, adding new HPC benchmark suites, and statistical significance testing, are preliminary thoughts for future work beyond the current project scope and
have no corresponding workplan entries.
Advanced Data Interaction and Display
• Simple web application: Turn the data processing scripts into a web app using Python,
using frameworks like Streamlit [32] or Dash [33]—which are open-source tools that allow
for the rapid creation of data-driven interfaces directly in Python. This avoids difficult frontend coding and focuses on showing data in an interactive way.
• Interactive kernel filtering and better visuals: Add ways to filter results so users can
look at the performance of just one calculation kernel. Also, use different patterns or
shading in charts to help users tell different LLVM versions apart more easily.
Giving the user more control
• Test subset selection: Create a feature that lets users choose exactly which tests to
run. This is meant to solve problems where output files are too large or tests take too
long, which reduces the work the computer has to do and makes the final report easier to
read.
• Letting users add their own tests: Allow users to upload their own code sets. The
system will then automatically create the configuration files and include that code in the
standard testing process.
Making the framework more pluggable
• Support for multiple test suite versions and parser formats: Make the data parser
more flexible so it can handle changes in data fields from different versions of a test suite
and support different parser formats. Or, set up strict version control for all dependencies.
• Adding new HPC evaluation frameworks: Besides the two current test suites, look into
adding OpenMP micro-benchmarks to measure the extra time used for thread management. Or, add the SOLLVE OMPVV test suite to check if the compiler follows programming
rules correctly.
Smart Analysis and Alerts
• Automatic performance change detection: After comparing versions, the system should
automatically calculate the differences in performance. It should list the kernels where
performance changed the most (either improved or regressed), filtered by a user-defined
significance threshold so that only meaningful changes are highlighted.
• Statistical significance testing: Allow the system to run the same test several times and
use basic statistics (like a t-test). The system will use this to decide if a small change in
speed is really caused by a change in the compiler or if it is just normal noise from the
computer system.
• Automated LLVM version monitoring: Set up a CRON job or GitLab CI [38] pipeline
that periodically checks the LLVM repository for new tags or commits. When a change is
detected, it automatically triggers the testing framework, enabling continuous and handsfree performance tracking without manual intervention.

Data Access and Ethics

All tools and test suites used in this project (like LLVM source code and the RAJA performance
suite) are public open-source software. There is no need to sign non-disclosure agreements or
ask for special data access.
This research only involves counting and analyzing the running time of computer programs.

approval is required.
